# 功能更新维护说明

这个目录用于维护功能新增、交互优化和重要修复的阶段性记录。

## 维护目标

- 避免把所有变更堆进一个超长文档。
- 让后续开发能快速知道最近做过什么、改了哪里、怎么验证。
- 把“需求描述”和“最终落地结果”分开，减少历史方案和当前实现混在一起。

## 目录规则

- 按年份建目录：`docs/updates/2026/`
- 按时间窗口建文件，默认一周一篇；如果某几天改动很多，也可以按 3 到 5 天单独开一篇。
- 文件名统一使用：`YYYY-MM-DD_to_YYYY-MM-DD.md`

示例：

- `docs/updates/2026/2026-06-01_to_2026-06-07.md`
- `docs/updates/2026/2026-06-08_to_2026-06-10.md`

## 什么时候新开文档

- 到了新的一周，直接开新文件。
- 单个文档已经明显过长，或包含多个独立主题时，提前结束当前文档。
- 大版本发布前，可以单独补一篇汇总，但不要替代日常更新记录。

## 每篇文档建议包含

1. 本周期目标
2. 已完成内容
3. 涉及文件
4. 验证方式
5. 已知限制或后续事项

## 模板

新建文档时可直接复制：

- `docs/updates/_template.md`
